大周礼时代网站仿制第一阶段：典故问答API

2026年7月12日 · IE-2007G

项目定位：本文介绍的是"合乎庄子"整体项目的第一阶段成果。完整的"合乎庄子"项目计划包含多个模块，本文所述的典故问答API只是其中的基础能力层。后续还将构建前端展示层、知识图谱层、以及面向特定场景的应用层。

项目简介#

《庄子》是道家哲学经典，全书寓言密布、典故分散。本项目想解决的问题：

能否用自然语言向 AI 提问庄子典故，利用AI精准定位典故，获取可靠译文？

技术路线：

  • 数据层：人工整理《庄子》原文中出现的典故列表，按篇章排列。全书 32 篇（《天下》篇不计入典故） 208 个典故结构化（原文 + 翻译 + 解读 + 关键词）
  • 检索层：BM25 全文检索 + 向量语义检索 + RRF 融合排序
  • 生成层：DeepSeek API 生成带引用的回答
  • 部署层：阿里云函数计算 FC（公网可访问）

为什么先做API#

整体项目计划中，典故问答API是基础能力层。它的设计原则：

  1. 无状态、可复用：API只负责检索和生成，不绑定任何前端。任何客户端（网页、小程序、命令行工具）都可以调用。
  2. 轻量可迁移：部署在函数计算上，按需伸缩，冷启动30秒内，后续请求毫秒级响应。
  3. 数据与逻辑分离：208条典故数据独立存储，检索算法可独立迭代，LLM可替换。

这样做的好处是：后续不管前端怎么变（是做成网页、小程序、还是嵌入到其他系统），后端API都不用动。

技术选型理由#

为什么用混合检索而不是纯向量？#

向量检索擅长语义理解（比如"庄子如何看待生死"），但对精确关键词（比如"庖丁解牛"）效果不如全文检索。

实测数据：

  • 纯BM25 Top-1准确率：83.3%
  • 纯向量 Top-1准确率：约75%
  • 混合检索（RRF融合）Top-1准确率：约90%

两者互补，效果显著。

为什么用阿里云FC而不是ECS？#

  • 按需付费：没人访问时不收费，适合演示项目
  • 免运维：不用管服务器、域名、HTTPS证书
  • 自动伸缩：突然有人访问时自动扩容

代价是冷启动慢（首次请求30秒），但对于演示场景可以接受。

为什么不用现成的RAG框架？#

LangChain、LlamaIndex这些框架确实能更快搭建RAG系统，但本项目选择从零实现，原因是：

  1. 数据量小：208条典故，不需要复杂的文档切片、向量数据库
  2. 可控性：自己实现可以精确控制每个环节（检索策略、prompt模板、结果格式）
  3. 学习价值：理解RAG的每个细节，比调API更有收获

技术栈总结#

层级 技术
数据处理 Python + JSON + SQLite FTS5
向量检索 BAAI/bge-m3 + numpy
LLM DeepSeek Chat API
Web框架 FastAPI + Uvicorn
部署平台 阿里云函数计算FC
前端 纯HTML/CSS/JavaScript
